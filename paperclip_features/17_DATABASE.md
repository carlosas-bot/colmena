# Database â€” embedded PGlite or external Postgres, Drizzle ORM, ~58 tables

## Investigation Summary

Paperclip uses PostgreSQL via Drizzle ORM with three deployment modes: embedded PostgreSQL (zero-config default), Docker PostgreSQL, and hosted PostgreSQL (Supabase, etc.). The database layer (~700 lines in `client.ts` alone) includes a sophisticated migration system with reconciliation, introspection, statement-level idempotency checking, and embedded Postgres lifecycle management. The schema spans 56 files defining ~58 tables with 44 migration files.

---

## 1. Database Modes

### 1.1 Embedded PostgreSQL (Default)

- Uses `embedded-postgres` npm package (PGlite)
- **Zero config** â€” no `DATABASE_URL` needed
- Auto-creates data directory: `~/.paperclip/instances/default/db/`
- Auto-initializes cluster with `--encoding=UTF8 --locale=C`
- Auto-creates `paperclip` database
- Auto-runs migrations on startup
- Data persists across restarts
- Port auto-discovery (starts at default, scans for free port)
- Handles stale `postmaster.pid` files (crash recovery)
- Adopts existing running instances on the same data directory

### 1.2 External PostgreSQL

- Set `DATABASE_URL` env var or config
- Docker Compose quickstart: PostgreSQL 17 on `localhost:5432`
- Hosted: Supabase, any PostgreSQL provider
- Connection pooling support: disable prepared statements for pgBouncer

### 1.3 Connection Resolution

```typescript
function resolveDatabaseTarget() {
  // 1. Check DATABASE_URL â†’ use external Postgres
  // 2. Check config for database settings â†’ use configured connection
  // 3. Fall back to embedded Postgres with auto data dir
}
```

## 2. ORM: Drizzle

### 2.1 Setup

```typescript
import { drizzle as drizzlePg } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema/index.js";

export function createDb(url: string) {
  const sql = postgres(url);
  return drizzlePg(sql, { schema });
}

export type Db = ReturnType<typeof createDb>;
```

- **Driver:** `postgres` (postgres.js â€” non-blocking, zero-dependency)
- **ORM:** `drizzle-orm` with `drizzle-orm/postgres-js` adapter

## 3. Migration System

### 3.1 Migration Files

- 44 SQL migration files in `packages/db/src/migrations/`
- Generated by Drizzle Kit: `pnpm db:generate`
- Applied on startup or manually: `pnpm db:migrate`
- Journal metadata in `migrations/meta/_journal.json`
- Statement breakpoints: `--> statement-breakpoint`

### 3.2 Migration Application

`applyPendingMigrations(url)`:

1. **Inspect** current state (`inspectMigrations`)
2. **Empty DB** â†’ run Drizzle's standard `migrate()`, then reconcile any gaps
3. **Non-empty DB without migration journal** â†’ error (unsafe to auto-migrate)
4. **Pending migrations** â†’ reconcile history first, then apply manually

### 3.3 Migration State Inspection

`inspectMigrations(url)` returns:

```typescript
type MigrationState =
  | { status: "upToDate"; tableCount; availableMigrations; appliedMigrations }
  | { status: "needsMigrations"; tableCount; availableMigrations; appliedMigrations; pendingMigrations;
      reason: "no-migration-journal-empty-db" | "no-migration-journal-non-empty-db" | "pending-migrations" }
```

### 3.4 Migration History Reconciliation

`reconcilePendingMigrationHistory(url)`:

Handles the case where migrations were already applied (tables exist) but the journal is out of sync. For each pending migration:

1. Read migration SQL content
2. Parse each statement
3. Check if already applied by inspecting DB state:
   - `CREATE TABLE` â†’ check `information_schema.tables`
   - `ADD COLUMN` â†’ check `information_schema.columns`
   - `CREATE INDEX` â†’ check `pg_class`
   - `ADD CONSTRAINT` â†’ check `pg_constraint`
4. If all statements already applied â†’ add to journal without re-running
5. If any statement not applied â†’ stop (don't skip ahead)

This is **remarkably sophisticated** â€” it can repair migration journals without data loss.

### 3.5 Manual Migration Application

`applyPendingMigrationsManually(url, pendingMigrations)`:

1. Order migrations by journal entry order
2. For each migration:
   - Read SQL content
   - Hash content (SHA-256) for deduplication
   - Check if hash already exists in journal
   - Split into statements at `-->  statement-breakpoint`
   - Execute each statement in a transaction
   - Record in migration journal

### 3.6 Migration Journal Schema

The `__drizzle_migrations` table supports multiple column layouts:
- Original: `(id, hash, created_at)`
- Newer: `(id, hash, name, created_at)`

The code handles both schemas, resolving migrations by hash and/or name.

## 4. Embedded Postgres Lifecycle

### 4.1 Startup

```typescript
async function ensureEmbeddedPostgresConnection(dataDir, preferredPort) {
  // 1. Check for stale postmaster.pid â†’ try to adopt existing instance
  // 2. If running PID found â†’ connect to existing instance
  // 3. If data dir exists but not running â†’ start embedded instance
  // 4. If fresh â†’ initialize cluster, then start
  // 5. Find free port (scan from preferred, max 20 lookahead)
  // 6. Ensure 'paperclip' database exists
}
```

### 4.2 Crash Recovery

- Detects stale `postmaster.pid` (PID no longer running)
- Removes stale pid file before starting
- Can adopt an already-running Postgres instance if data dirs match

## 5. Query Patterns

### 5.1 Service-DB Interaction

All services receive a `Db` instance and use Drizzle's query builder:

```typescript
export function myService(db: Db) {
  return {
    list: () => db.select().from(myTable).where(eq(myTable.companyId, id)),
    create: (data) => db.insert(myTable).values(data).returning(),
    update: (id, data) => db.update(myTable).set(data).where(eq(myTable.id, id)).returning(),
  };
}
```

### 5.2 Transaction Usage

Multi-table operations use `db.transaction()`:
```typescript
return db.transaction(async (tx) => {
  await tx.update(table1).set(...);
  const result = await tx.insert(table2).values(...).returning();
  return result;
});
```

### 5.3 Raw SQL

Complex queries use `sql` template literal:
```typescript
sql<number>`coalesce(sum(${costEvents.costCents}), 0)::int`
```

## 6. Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Embedded Postgres as default | Zero-config developer experience. No Docker, no setup. Just `pnpm dev`. |
| Drizzle ORM | Type-safe, lightweight, good DX. No heavy abstraction like Prisma. |
| `postgres` driver (not `pg`) | Zero dependencies, faster, better TypeScript support |
| UUID primary keys everywhere | No sequential IDs to leak information. Works across distributed systems. |
| All timestamps with timezone | Correct handling of UTC across different server locations |
| JSONB for flexible fields | Adapter configs, metadata, context snapshots â€” all need schema-free storage |
| Custom migration reconciliation | Handles messy real-world scenarios: partial migrations, journal drift, statement-level idempotency |
| 56 schema files (not one big file) | Modular, grep-friendly, clear ownership per domain |
| Company-scoped FK on every entity | Enforces data isolation at the schema level |

## 7. Observations for Colmena

1. **Embedded Postgres is a killer DX feature** â€” zero-config local development. Absolutely worth adopting.

2. **The migration reconciliation system is overengineered** for most use cases but handles real production scenarios (partial deploys, journal corruption). For Colmena MVP, standard Drizzle migrations are sufficient.

3. **Drizzle ORM is a good choice** â€” lightweight, type-safe, composable queries. Better than Prisma for this use case (more control, less magic).

4. **56 schema files is a lot** â€” but each is small and focused. This organization works well with TypeScript's module system.

5. **UUID everywhere** is the right default for a multi-tenant system. Avoids sequential ID leaks and works well with distributed setups.

6. **JSONB is used heavily** (adapterConfig, runtimeConfig, metadata, contextSnapshot, payload, etc.). This trades query performance for schema flexibility â€” acceptable for typical Paperclip workloads.

7. **For Colmena MVP**, start with: embedded Postgres, Drizzle ORM, standard migrations (`drizzle-kit push` for development, `migrate()` for production). Add reconciliation and backup later.

8. **The backup system** (`runDatabaseBackup`/`runDatabaseRestore`) is a production necessity. Plan for it early â€” even if the MVP doesn't implement it, schema design should be backup-friendly.

9. **The `Db` type** is passed to every service as a dependency injection pattern. This makes testing easy (pass a mock/test DB) and keeps services stateless.
